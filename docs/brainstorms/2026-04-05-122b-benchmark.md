# Brainstorm: Benchmark Qwen3.5-122B-A10B on M5 Max 128GB

**Bead:** sylveste-e25
**Date:** 2026-04-05

## Context

The interfer routing table lists three tiers:
- C1: Qwen3.5-9B (5GB, ~60-80 tok/s) — trivial tasks
- C2: Qwen3.5-35B-A3B (18GB, ~86 tok/s) — routine tasks
- C3: Qwen3.5-122B-A10B (~65GB, pending benchmark) — moderate tasks

We also have the 397B model via flash-moe (1.98 tok/s SSD-streamed). The 122B
is the critical gap: if it can do 15+ tok/s with acceptable quality, it fills
the space between the fast 35B and the slow-but-powerful 397B.

## Two Benchmarks

### Benchmark A: Qwen3.5-122B-A10B via MLX (new)

**Model:** `mlx-community/Qwen3.5-122B-A10B-4bit` (69.6GB download)
**Method:** mlx-lm standard inference — model fits entirely in RAM (65GB model + 10GB OS + ~53GB KV headroom)
**Expected:** 15-40 tok/s (MoE with 10B active, similar architecture to 35B-A3B which does 86 tok/s with 3B active)

**Metrics to capture:**
- tok/s (generation), prompt tok/s (prefill)
- TTFT (time to first token)
- Peak memory usage (Metal active + peak)
- Quality: perplexity on wikitext-2 subset, coding task accuracy

**Test prompts:**
1. Short coding task: "Write a Python function to find the longest common subsequence"
2. Code review: paste 100-line function, ask for review
3. Instruction following: multi-step task decomposition
4. System prompt + conversation: simulate Clavain routing context

### Benchmark B: Qwen3.5-397B quality eval (existing infrastructure)

**Model:** flash-moe with Q3 experts + cache-io-split 4 (1.98 tok/s)
**Method:** Direct binary invocation via existing benchmark scripts
**Goal:** Not throughput (already measured) — quality comparison against 122B

**Same test prompts as Benchmark A** for apples-to-apples quality comparison.

## Memory Budget Analysis

```
128GB total
- 10GB: macOS + system
- 65GB: Qwen3.5-122B-A10B 4-bit weights
- 53GB: Available for KV cache + headroom
```

At 4-bit KV (kv_bits=4, proven free lunch from Experiment 2):
- Each KV entry: ~half the standard size
- 53GB headroom should support 8K+ context comfortably

The 122B model CANNOT coexist with the 397B flash-moe model in RAM.
Benchmarks must be run sequentially, not simultaneously.

## Key Questions

1. **Does 122B fit?** 65GB model + OS should leave ~53GB. If Metal overhead
   pushes total past 118GB, we'll see OOM or degraded performance.

2. **What's the MoE speedup factor?** 122B has 10B active of 122B total.
   The 35B-A3B (3B active of 35B) does 86 tok/s. Naive scaling: 
   10B/3B × penalty factor ≈ 3.3× slower → ~26 tok/s. But memory bandwidth
   may be the bottleneck, not compute.

3. **Quality gap: 122B vs 397B?** If the 122B matches 397B on coding tasks,
   the 397B's 1.98 tok/s SSD streaming becomes hard to justify for C3 tasks.

## Success Criteria

- [ ] 122B loads and generates without OOM
- [ ] Benchmark results TSV with tok/s, TTFT, memory for both models
- [ ] Quality comparison on 4+ coding tasks
- [ ] Recommendation: which model for C3 tier in the routing table
