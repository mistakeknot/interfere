# Reflection: Benchmark Qwen3.5-122B-A10B on M5 Max 128GB

**Bead:** sylveste-e25
**Date:** 2026-04-05

## What happened

Benchmarked Qwen3.5-122B-A10B (MLX, 4-bit, 65GB) and Qwen3.5-397B (flash-moe,
Q3+cis4, SSD-streamed) on 4 C3-level coding tasks. The 122B was dramatically
slower than predicted (2.99 tok/s vs expected 15-40 tok/s).

## What we learned

1. **MoE active-param scaling doesn't predict throughput.** The 35B-A3B at 86 tok/s
   with 3B active suggested ~26 tok/s for 122B-A10B with 10B active. Actual: 2.99
   tok/s. The bottleneck is memory bandwidth, not compute — reading routing decisions
   across 65GB of weight tensors saturates the M5 Max's 614 GB/s bandwidth. Scaling
   models must account for total weight size, not just active parameter count.

2. **RAM-resident doesn't mean fast for large MoE.** The 122B fits entirely in RAM
   but is only 2.8× faster than the 397B streaming from SSD. Meanwhile the 397B
   leaves 122 GB free for the model fleet, KV caches, and OS. The memory-cost-adjusted
   performance is overwhelmingly in favor of flash-moe.

3. **Quality is equivalent between 122B and 397B on coding tasks.** On all 4 benchmarks
   the models produced indistinguishable quality. This kills the argument for the 122B
   as a C3 model — the 397B gives the same quality at 1/11th the RAM cost.

4. **FlashMoeWorker had the wrong port flag (`--port` vs `--serve`).** This was caught
   during the benchmark, not during the previous sprint's test fixes. The unit tests
   use a fake server and never invoke the real binary, so the flag name mismatch was
   invisible. End-to-end smoke tests with the real binary are irreplaceable.

## What we'd do differently

Before downloading a 70GB model, we should have first checked whether the
theoretical performance justified the memory cost. A back-of-envelope calculation
based on memory bandwidth (65GB model ÷ 614 GB/s = 106ms minimum per token for
a full weight scan) would have predicted ~9 tok/s maximum, which is already
marginal. The actual 2.99 tok/s reflects additional overhead from routing,
attention, and KV cache operations.

## Routing table update

C3 tier updated from `qwen3.5-122b-a10b-4bit (pending)` to
`flash-moe:qwen3.5-397b (Q3+cis4, ~1 tok/s) OR cloud escalation`.
